# Технологический стек проекта (Production / Реальная 1С)

Основываясь на целевой архитектуре проекта, при подключении к реальной системе 1С технологический стек расширяется и разделяется на пять основных компонентов: Фронтенд (Next.js), Бэкенд-оркестратор (Next.js API), микросервис извлечения данных (GLiNER), микросервис интеграции (MCP Server) и непосредственно инфраструктуру 1С.

Ниже представлено подробное разделение Production-стека.

## 1. Фронтенд (Frontend / UI)

Фронтенд построен на базе **Next.js 14.2 (App Router)** и **React 18**. Исходя из того, как он работает, его можно разделить на две части:

### А. Использование в браузере (Стандартные Web-технологии)
Эта часть отвечает за отображение интерфейса, стилизацию и логику общения с сервером по стандартным веб-протоколам.
* **Язык программирования:** TypeScript.
* **Стилизация:** Tailwind CSS, PostCSS.
* **UI-библиотека:** Fluent UI React v9 (`@fluentui/react-components`) — официальная библиотека от Microsoft, позволяющая интерфейсу выглядеть как нативная часть системы Windows/Office.
* **Иконки:** Lucide React.
* **Управление состоянием чата и ИИ:** Vercel AI SDK (клиентский хук `useChat`).

### Б. Использование в качестве плагина (Outlook Add-in)
Эта часть отвечает за то, чтобы веб-приложение могло «жить» внутри клиента Microsoft Outlook и взаимодействовать с его контекстом.
* **API взаимодействия:** Office.js (`@microsoft/office-js`). Извлекает данные письма (отправитель, тема, текст), поддерживает закрепление панели (Pinning).
* **Манифест:** `manifest.xml` — XML-файл конфигурации Outlook.
* **Адаптация окружения:** Кастомные патчи в `layout.tsx` для обхода конфликтов навигации Office.js и Next.js.

---

## 2. Бэкенд (Backend / AI Orchestrator)

Бэкенд реализован в том же проекте Next.js с использованием Route Handlers. Он выступает в роли «моста» между фронтендом, нейросетью и микросервисами.
* **Маршрутизация API:** Next.js Route Handlers (`/api/chat`).
* **AI Оркестрация:** Vercel AI SDK (`streamText`, обработка `tools`).
* **LLM Провайдер:** OpenRouter API (доступ к моделям OpenAI, Anthropic и др. с поддержкой Tool Calling).
* **Связь с микросервисами:** Вызов GLiNER для преданализа и отправка команд в MCP Server для работы с 1С.

---

## 3. Микросервис извлечения данных (GLiNER / NER)

Отдельный микросервис для быстрого и локального извлечения структурированной информации (именованных сущностей: имена, названия компаний, контакты, суммы, продукты) из текста писем.
* **Среда выполнения:** Python.
* **Web-фреймворк:** FastAPI (или Flask) для предоставления легковесного REST API.
* **Модель:** GLiNER (Generalist and Lightweight Model for Named Entity Recognition). Легковесная нейросеть, работающая без вызова тяжелых облачных LLM.
* **Функция:** Обогащение данных для предзаполнения полей в 1С и экономия токенов основной LLM.

---

## 4. Микросервис интеграции (MCP Server)

Сервис-посредник, инкапсулирующий логику взаимодействия с 1С и векторный поиск (RAG).
* **Среда выполнения:** Node.js, TypeScript.
* **Сетевой протокол:** Model Context Protocol (MCP) SDK. Стандарт, превращающий микросервис в сервер инструментов для LLM.
* **Интеграция с реальной 1С:** 
  * Взаимодействие происходит по **HTTP/REST** или протоколу **OData**.
  * Используются HTTP-клиенты (например, `axios` или встроенный `fetch`) для отправки запросов к опубликованным веб-сервисам 1С (создание сделок, обновление статусов).
  * **Безопасность:** Авторизация через Basic Auth, JWT-токены или OAuth, настроенные на стороне веб-сервера 1С.
* **Семантический поиск (Production RAG):**
  * **Фреймворк:** LangChain.
  * **Векторная БД (Вместо In-memory):** Использование полноценной персистентной СУБД для хранения векторов. Это может быть **PostgreSQL с расширением pgvector**, **Qdrant**, **Milvus** или облачные решения (Pinecone).
  * **Синхронизация данных:** Регулярная (cron) или событийная (через Webhooks от 1С) синхронизация данных контрагентов и сделок из 1С в векторную БД.
  * **Эмбеддинги:** Опциональный переход с публичного HuggingFace API на локальные модели (например, через Ollama/TEI) для соответствия требованиям NDA и информационной безопасности.

---

## 5. Учетная система (Сторона 1С:CRM УТиВСК)

Реальная система 1С, выступающая мастер-системой (Single Source of Truth) для всех бизнес-данных.
* **Платформа:** 1С:Предприятие 8.3.
* **Конфигурация:** 1С:CRM УТиВСК (Управление торговлей и взаимоотношениями с клиентами).
* **Веб-сервер для публикации баз:** **IIS (Internet Information Services)** от Microsoft или **Apache**. Необходим для того, чтобы база 1С могла принимать внешние HTTP-запросы.
* **СУБД 1С:** MS SQL Server или PostgreSQL (для хранения всех учетных данных).
* **API интерфейс:**
  * Либо стандартный OData интерфейс 1С.
  * Либо специально разработанные **HTTP-сервисы** внутри конфигурации 1С:CRM УТиВСК для оптимизации запросов (например, кастомные роуты для поиска контрагентов, создания Интересов (Сделок) и привязки взаимодействий).

---

## 6. Архитектура и взаимодействие компонентов

Взаимодействие компонентов в системе происходит по следующему сценарию (от получения письма до создания записи в 1С):

1. **Инициализация (Outlook -> Frontend)**
   * Пользователь открывает письмо в Outlook и нажимает на плагин.
   * `Office.js` на Фронтенде извлекает метаданные письма (тема, отправитель, текст) и передает их в пользовательский интерфейс (UI).

2. **Первичный анализ и извлечение данных (Frontend -> Backend -> GLiNER)**
   * Фронтенд отправляет текст письма на Бэкенд (`/api/chat`).
   * *Опционально:* Бэкенд делает быстрый синхронный запрос к микросервису **GLiNER**, чтобы моментально выделить сущности (имена, названия компаний, суммы).
   * Извлеченные структурированные данные возвращаются на Бэкенд для обогащения контекста.

3. **Оркестрация и принятие решений (Backend -> LLM)**
   * Бэкенд формирует промпт (содержащий текст письма, извлеченные GLiNER данные и доступные инструменты) и отправляет его в основную **LLM (OpenRouter)**.
   * Модель анализирует запрос и решает, какие действия (Tools) необходимо выполнить (например, найти существующего клиента или создать новый Интерес).

4. **Выполнение инструментов и поиск (Backend -> MCP Server -> Vector DB / 1C)**
   * Если LLM решает вызвать инструмент, Бэкенд обращается к **MCP Server**.
   * MCP Server выполняет запрошенную операцию:
     * *При поиске (RAG):* Вычисляет эмбеддинги запроса и ищет релевантные сделки в **Векторной БД** (Qdrant/PgVector).
     * *При чтении данных:* Делает HTTP-запрос в **1С:CRM УТиВСК** для получения актуального статуса клиента или сделки.
   * Результаты возвращаются обратно в LLM для формирования итогового ответа или предложения пользователю.

5. **Подтверждение и запись в 1С (User -> Frontend -> Backend -> MCP Server -> 1C)**
   * LLM возвращает на Фронтенд интерактивные кнопки (например, "Создать Интерес в 1С").
   * Пользователь в UI подтверждает действие.
   * Фронтенд отправляет подтверждение на Бэкенд, который маршрутизирует команду в **MCP Server**.
   * MCP Server формирует POST/PUT запрос по REST API (или OData) и отправляет его на опубликованный веб-сервер **1С:CRM УТиВСК**.
   * 1С обрабатывает запрос, создает документ и возвращает ID созданной сущности.
   * Успешный статус отображается пользователю в интерфейсе Outlook.
